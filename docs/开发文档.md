# 开发文档（架构与扩展指南）

本文档面向想要**理解 bot.py 内部结构**或**二次开发**的开发者。
面向终端用户的使用说明请看根目录的 [README.md](../README.md)。

---

## 一、项目结构

```
Bot/
├── bot.py                       # 主程序（全部核心逻辑，单文件）
├── config.json                  # 配置文件（.gitignore，含真实 QQ 号）
├── data.json                    # 运行时数据（.gitignore，自动生成）
├── bot.log                      # 运行日志（.gitignore）
├── requirements.txt             # Python 依赖
├── examples/
│   └── config.example.json      # 配置模板（不含真实号，可提交）
├── scripts/
│   └── linux/bot.service        # Linux systemd 服务文件
├── 启动.bat / 停止.bat          # Windows 启动 / 停止脚本
├── 后台运行.vbs                  # Windows 后台静默运行
├── start.sh                     # Linux 启动 / 停止 / 守护脚本
├── README.md                    # 用户文档
└── docs/
    └── 开发文档.md              # 本文件
```

> 设计取舍：项目刻意保持**单文件**结构（`bot.py`），方便用户「整个文件夹打包就能搬走」。
> 因此代码用清晰的注释分区 + 命令注册表来组织，而不是拆成多包。

## 二、运行流程（数据流）

```
            ┌──────────┐   正向 WebSocket    ┌────────────┐
            │  NapCat  │ ◀──────────────────▶ │   bot.py   │
            │ (协议端)  │   ws://127.0.0.1:3001 │  (大脑)    │
            └──────────┘                      └────────────┘
                 ▲                                  │
                 │ 登录 QQ、收发消息、踢人/禁言         │ call_api()
                 │                                  ▼
              QQ 服务器                       事件分发 dispatch()
```

1. **`run()`** 主循环用 `websockets` 连接 NapCat，连上后缓存机器人 QQ（`BOT_QQ`）。
2. 每收到一条消息 `asyncio.create_task(_safe_dispatch(raw))` 异步分发，**不阻塞接收循环**（否则 API 响应会和自己死锁）。
3. **`dispatch(raw)`** 解析 JSON：
   - 带 `echo` 的是 API 响应 → 匹配 `PENDING` 里的 Future，交给对应的 `call_api()` 调用者。
   - 带 `post_type` 的是事件 → 交给 `handle_message` / `handle_notice` / `handle_request`。
4. **`call_api()`** 是 OneBot 11 API 调用的核心：发 `{action, params, echo}`，用 `echo` 匹配异步响应，带超时。

## 三、命令注册表（核心机制）

群命令采用**表驱动分发**，定义在 `bot.py` 的 `_GROUP_COMMAND_TABLE`：

```python
# (别名..., 处理函数, 权限等级, 拒绝提示)
_GROUP_COMMAND_TABLE = (
    (("mute", "禁言"),            cmd_mute,     None,        None),
    (("group", "grp", "生效群"),  cmd_group,    "botadmin",  "⛔ 仅Owner/Bot Admin可管理生效群。"),
    (("black", "拉黑"),           cmd_black,    "owner",     "⛔ 仅Owner可使用此命令。"),
    # ...
)

_GROUP_ALIASES = {
    alias: (handler, tier, deny)
    for aliases, handler, tier, deny in _GROUP_COMMAND_TABLE
    for alias in aliases
}
```

`handle_message` 里的群命令分发因此只剩几行：

```python
entry = _GROUP_ALIASES.get(cmd)
if entry is None:
    await send_group_text(group_id, f"🤔 未知指令 /{cmd} ...")
    return
handler, tier, deny_msg = entry
if tier == "owner"    and not is_owner(user_id):     ...
if tier == "botadmin" and not is_bot_admin(user_id): ...
await handler(event, rest)
```

**权限等级（tier）含义：**

| tier | 谁能用 | 说明 |
|------|--------|------|
| `None` | 群主 / 管理员 / Owner / Bot Admin | 已由上方「权限闸门」保证，无需再判 |
| `"botadmin"` | 仅 Owner / Bot Admin | 如 `/group` `/yes` `/no` `/pending` |
| `"owner"` | 仅 Owner | 如 `/black` `/unblack` |

> ⚠️ **关于「权限闸门」**：`handle_message` 在进入命令表之前已经执行过
> `if not (is_admin(event) or is_bot_admin(user_id)): 拒绝并 return`。
> 因此 `tier=None` 的命令无需再做权限判断 —— 旧版本里每条命令重复写的
> `if not admin and not is_bot_admin(...)` 属于**不可达死代码**，已在优化中移除。

**所有群命令处理器统一签名为 `async def handler(event, rest)`**（`rest` 是命令前缀后的剩余文本）。
即使是无需参数的 `/sid`、`/pending`，签名也是 `(event, rest="")`，以保持分发统一。

## 四、权限模型（三层）

| 身份 | 来源 | 能做什么 |
|------|------|----------|
| **Owner** | `config.json` 的 `super_admins` | 所有命令；不会被封禁/禁言/踢出；唯一能管理黑名单和 Bot Admin |
| **Bot Admin** | `config.json` 的 `admins`（由 Owner 私聊 `/admin` 管理） | 除 `/black` `/unblack` `/admin` 外的所有命令 |
| **群主 / 管理员** | QQ 群内角色（受 `allow_owner` / `allow_admin` 开关） | 基本群管理命令 |
| **黑名单用户** | `config.json` 的 `banned_users` | 即使是群主/管理员也不能用命令 |

判定函数：

- `is_owner(user_id)` — 是否 Owner
- `is_bot_admin(user_id)` — Owner **或** Bot Admin
- `is_admin(event)` — Owner，或（未被拉黑的）群主/管理员

## 五、配置与数据存储

### `config.json`（可热重载：群里 `/reload`）

| 键 | 说明 |
|----|------|
| `ws_url` | NapCat 正向 WebSocket 地址 |
| `super_admins` | Owner QQ 号列表 |
| `admins` | Bot Admin QQ 号列表 |
| `banned_users` | 黑名单（禁用命令 + 拦截加好友/加群/邀请） |
| `allow_owner` / `allow_admin` | 是否允许 QQ 群主 / 管理员使用命令 |
| `allowed_groups` | 生效群（空 = 全部群） |
| `command_prefix` | 命令前缀（默认 `/`） |
| `reconnect_delay` | 断线重连间隔（秒） |
| `reapply_interval` | 永久禁言续期间隔（秒，默认每天） |
| `mute_max_seconds` | QQ 单次禁言上限（30 天，一般不改） |
| `welcome_message` / `leave_message` | 全局欢迎/退群消息模板 |
| `reject_non_owner_invites` | 自动拒绝非 Owner 的群邀请 |
| `notify_dm` | 有人私聊 bot 时是否通知 Owner |
| `auto_leave_check_interval` | 自动退群巡检间隔（秒） |

> 配置文件允许写以 `_` 开头的说明性字段（如 `"_说明": "..."`），
> `load_config()` 会自动过滤它们，不影响运行。

### `data.json`（运行时自动生成）

| 键 | 结构 | 说明 |
|----|------|------|
| `bans` | `{ "群号_QQ号": {reason, expire, set_at, set_by, ...} }` | 封禁记录，`expire=0` 为永久 |
| `mutes` | 同上 | 禁言记录 |
| `whitelist` | `{ 群号(str): [QQ号, ...] }` | 加群白名单 |
| `whitelist_enabled` | `[群号, ...]` | 已开启白名单模式的群 |
| `welcome_msgs` / `leave_msgs` | `{ 群号(str): "模板" }` | 每群独立的欢迎/退群消息（覆盖全局） |

写入采用 `tmp + os.replace()` 原子替换，避免写一半断电损坏文件。

## 六、后台任务

启动时 `run()` 用 `asyncio.create_task` 起两个常驻协程：

- **`background_tasks()`**：按 `reapply_interval` 周期运行
  ① 给所有「永久禁言」重新上 30 天（QQ 单次上限 30 天，靠续期实现永久）；
  ② 清理已过期的限时封禁/禁言记录。
- **`auto_sync_groups()`**：按 `auto_leave_check_interval` 周期运行（仅当 `allowed_groups` 非空）
  ① 机器人在群、但群不在生效列表 → 自动退群；
  ② 生效列表里有、但机器人已不在该群 → 自动移除。

另有 `stdin_reloader_loop()` 跑在守护线程里：在 CMD 输入 `r` + 回车 → `os._exit(2)`，
由启动脚本检测退出码 2 后**立即重启**进程（实现不重启脚本的热重载）。

## 七、启动脚本与退出码

`启动.bat`（Windows）和 `start.sh`（Linux）都是「保活循环」，根据 `bot.py` 的退出码决定下一步：

| 退出码 | 含义 | 启动脚本行为 |
|--------|------|--------------|
| `0` | 正常停止（Ctrl+C） | 退出，不重启 |
| `2` | 请求热重载（输入 `r`） | **立即重启**，加载最新 `bot.py` |
| `3` | 致命错误 | 等 5 秒后重启 |
| 其他 | 崩溃 | 等 5 秒后重启 |

> 在 CMD 里输入 `r` 回车 = 热重载；群里发 `/reload` = 只刷新 `config.json`（不重启进程）。

## 八、如何添加一个新命令

以加一个 `/weather <城市>`（任意特权成员可用）为例：

1. **写处理器**（统一签名 `(event, rest)`）：
   ```python
   async def cmd_weather(event, rest):
       group_id = event["group_id"]
       city = rest.strip() or "北京"
       await send_group_text(group_id, f"🌤 {city}：晴（示例）")
   ```

2. **注册到命令表**（加一行，含中英文别名、权限等级）：
   ```python
   (("weather", "天气"), cmd_weather, None, None),
   ```
   - 任意特权成员 → `None`
   - 仅 Owner/Bot Admin → `"botadmin"`
   - 仅 Owner → `"owner"`（并填第 4 列拒绝提示）

3. **如需私聊也能用**：在 `handle_message` 的 `private` 分支里加一个 `if cmd in (...)` 分支（私聊命令分发表目前是手写的，未走表）。

4. **更新帮助文本** `HELP_SHORT` / `HELP_FULL`，以及 `README.md` 的命令表。

完成。无需改动分发逻辑。

## 九、外部 API 与可选依赖

- **GitHub 卡片**（`/github`）：调 `api.github.com`，用 Pillow 画卡片。依赖 `requests` + `Pillow`。
- **B 站卡片**（`/bilibili`）：调 `api.bilibili.com`，下载封面 + Pillow 画卡片。依赖 `requests` + `Pillow`。
- 这两者在 `loop.run_in_executor(None, ...)` 里跑（线程池），避免阻塞事件循环。
- 字体走模块级缓存 `_load_cjk_font(size)`：首次加载微软雅黑等 TTF 后缓存，避免每次画卡都读盘。

> `requests` / `Pillow` 是可选依赖，缺失时 `/github`、`/bilibili` 会提示安装，不影响其它功能。

## 十、NapCat API 兼容性说明

不同 NapCat 版本的 API 名称不一致，代码对关键操作都**按优先级尝试多个候选 API**，命中即停：

- 群公告：`_send_group_notice` → `send_group_notice` → `set_group_notice`
- 精华：`set_essence_msg` → `_set_essence_msg` → `set_essence_message`
- 黑名单：`set_blacklist` → `add_blacklist` → `block_user` → ...
- 群邀请回群：`invite_group_member` → `send_group_invite` → ...

因此即使某些 API 在当前 NapCat 版本不可用，功能也会优雅降级并给出提示，而不是崩溃。

## 十一、近期优化记录

- **命令分发重构**：群命令由 120+ 行的 `if/elif` 链改为**表驱动**（`_GROUP_COMMAND_TABLE`），
  移除了约 14 处因「权限闸门」存在而不可达的重复权限判断（死代码）。新增命令现在只需加一行。
- **性能**：`/github`、`/bilibili` 画卡片的字体改为模块级缓存（`_FONT_CACHE`），不再每次读盘。
- **兼容性**：`asyncio.get_event_loop()` → `asyncio.get_running_loop()`（Python 3.10+ 已弃用前者）。
- **代码整洁**：删除函数内重复的 `import re` / `from datetime import datetime`，统一为顶层导入；
  `load_db()` 读取失败时的兜底字典补全了 `welcome_msgs` / `leave_msgs` 键。
- **新功能**（同期）：私聊命令 `/group` `/friend` `/delfriend` `/reply` `/name` `/leave`，
  私聊通知 Owner（`notify_dm`），`赞我` 支持私聊 + 失败冷却。
